# tb_Neuromorphic_X2_wb_beh Testbench README

This document explains the testbench in:

```text
tb_Neuromorphic_X2_wb_beh.v
```

The testbench verifies and demonstrates the Wishbone-facing behavior of:

```text
Neuromorphic_X2_wb_beh.v
```

It drives configuration packets, SET/RESET/READ command packets, runtime
reconfiguration, and Wishbone reads that capture TDC-style output words.

## DUT Instance

The testbench instantiates:

```verilog
Neuromorphic_X2_wb_beh dut
```

with shorter simulation delays:

```verilog
.READ_DELAY(8),
.PROGRAM_DELAY(12),
.COMPUTE_DELAY(10),
.CONFIG_WRITES(3)
```

These reduced delays make simulation faster while preserving the ordering and
handshake behavior of the model.

## Clock And Reset

The Wishbone clock is generated by:

```verilog
always #5 wb_clk_i = ~wb_clk_i;
```

This gives a 10 ns clock period in simulation.

Reset is applied by `apply_reset()`:

```verilog
wb_rst_i = 1'b1;
init_bus();
repeat (5) @(posedge wb_clk_i);
@(posedge wb_clk_i);
wb_rst_i <= 1'b0;
repeat (2) @(posedge wb_clk_i);
```

During reset, the bus is driven idle.

## Wishbone Bus Signals

The testbench drives:

```verilog
wbs_stb_i
wbs_cyc_i
wbs_we_i
wbs_sel_i
wbs_dat_i
wbs_adr_i
```

and observes:

```verilog
wbs_dat_o
wbs_ack_o
```

All transactions use:

```verilog
ADDR_MATCH = 32'h3000_0004;
wbs_sel_i  = 4'hF;
```

## ACK Monitor

The testbench includes an ACK-width checker:

```verilog
if (ack_prev && wbs_ack_o) begin
  $display("%0t ERROR: wbs_ack_o stayed high for more than one clock", $time);
  $finish;
end
```

This checks that `wbs_ack_o` does not stay high on two consecutive clock edges.
It is a handshake check only. The testbench does not compare read data against
expected TDC values in the active stimulus flow.

## Packet Helper

The helper function:

```verilog
make_packet(mode, row, col, status_read, full_row, data_byte)
```

returns:

```verilog
{mode, row, col, status_read, full_row, 10'd0, data_byte}
```

The fields are:

```text
[31:30] mode
[29:25] row
[24:20] column
[19]    status_readback
[18]    full_row
[17:8]  zero in this helper
[7:0]   data_byte
```

Modes:

```verilog
2'b00 = RESET
2'b01 = READ
2'b10 = COMPUTE
2'b11 = SET
```

## Wishbone Write Task

The task:

```verilog
wb_write32(ADDR_MATCH, data_word);
```

performs a 32-bit Wishbone write.

Sequence:

```verilog
@(posedge wb_clk_i);
wbs_adr_i <= addr;
wbs_dat_i <= data_word;
wbs_sel_i <= 4'hF;
wbs_we_i  <= 1'b1;
wbs_cyc_i <= 1'b1;
wbs_stb_i <= 1'b1;

wait(wbs_ack_o);

@(posedge wb_clk_i);
init_bus();
```

Meaning:

1. Drive address, data, select, write-enable, cycle, and strobe on a positive
   clock edge.
2. Wait until the DUT raises `wbs_ack_o`.
3. On the next positive edge, return the bus to idle.

The testbench also defines:

```verilog
wb_write(data_word);
```

which simply calls:

```verilog
wb_write32(ADDR_MATCH, data_word);
```

## Wishbone Read Task

The task:

```verilog
wb_read32(ADDR_MATCH, rdata);
```

performs a 32-bit Wishbone read.

Sequence:

```verilog
@(posedge wb_clk_i);
wbs_adr_i <= addr;
wbs_dat_i <= 32'h0000_0000;
wbs_sel_i <= 4'hF;
wbs_we_i  <= 1'b0;
wbs_cyc_i <= 1'b1;
wbs_stb_i <= 1'b1;

wait(wbs_ack_o);
data_word = wbs_dat_o;

@(posedge wb_clk_i);
init_bus();
```

Meaning:

1. Drive a read request on the Wishbone bus.
2. Wait for the DUT to ACK.
3. Capture `wbs_dat_o` into the output argument.
4. Drop the bus on the next positive clock edge.

The active testbench uses `wb_read32` directly:

```verilog
wb_read32(ADDR_MATCH, rdata);
```

Then the value can be printed or decoded:

```verilog
$display("%0t WB_READ data=%08h", $time, rdata);
```

## Initial Configuration

The testbench writes three configuration packets immediately after reset:

```verilog
TARGET_SET1   = 16'h0300;
TARGET_SET2   = 16'h0200;
TARGET_RESET1 = 16'h0040;
TARGET_RESET2 = 16'h0080;
```

The helper task:

```verilog
write_configuration();
```

performs:

```verilog
wb_write({TARGET_SET2, TARGET_SET1});
wb_write({TARGET_RESET2, TARGET_RESET1});
wb_write({2'b01, 3'b000, 7'd32, 10'd3, 10'd3});
```

The initial SET/RESET midpoints are:

```verilog
SET midpoint   = 14'h0280;
RESET midpoint = 14'h0060;
```

## Runtime Reconfiguration

The testbench defines:

```verilog
RECONFIG_PACKET = {2'b11, 5'd0, 5'd0, 1'b0, 1'b0, 1'b1, 9'd0, 8'h00};
```

This is the runtime reconfiguration trigger:

```text
mode = SET
row  = 0
col  = 0
bit17 = 1
```

After this write, the next three writes are treated as configuration packets.

The second configuration uses:

```verilog
TARGET2_SET1   = 16'h0500;
TARGET2_SET2   = 16'h0400;
TARGET2_RESET1 = 16'h0100;
TARGET2_RESET2 = 16'h0180;
```

The helper task:

```verilog
write_configuration_2();
```

performs:

```verilog
wb_write({TARGET2_SET2, TARGET2_SET1});
wb_write({TARGET2_RESET2, TARGET2_RESET1});
wb_write({2'b01, 3'b000, 7'd40, 10'd4, 10'd5});
```

## Active Stimulus Flow

The active sequence is:

1. Reset the DUT.
2. Send the initial three configuration writes.
3. Program several cells with SET commands.
4. Send READ commands for selected cells.
5. Wait enough clock cycles for model delays.
6. Perform `wb_read32` transactions to capture TDC result words.
7. Program one cell with RESET.
8. Read that RESET-programmed cell.
9. Run Compute Test 1 before runtime reconfiguration.
10. Trigger runtime reconfiguration.
11. Send the second three configuration writes.
12. Repeat a similar SET/READ/RESET flow after reconfiguration.
13. Run Compute Test 2 after runtime reconfiguration.

Example from the active flow:

```verilog
wb_write(make_packet(2'b11, 5'd1, 5'd3, 1'b0, 1'b0, 8'hA5));
wb_write(make_packet(2'b01, 5'd1, 5'd3, 1'b0, 1'b0, 8'h12));
...
wb_read32(ADDR_MATCH, rdata);
```

Meaning:

1. SET row `1`, column `3` with program byte `8'hA5`.
2. Send a READ command for row `1`, column `3` with read byte `8'h12`.
3. Later perform a Wishbone read to capture the generated response.

## Active Compute Functionality

The updated behavioral model returns 32 Wishbone read words after one compute
operation.

The compute command is sent as three packets. The selected compute column is
taken from the third compute packet:

```verilog
selected_col = third_compute_packet[24:20];
```

The output order is:

```text
selected column first,
then columns 0 to 31 in order,
skipping the selected column
```

The first output word contains the computed result for the selected column:

```verilog
rdata = {13'd0, selected_col[4:0], compute_tdc[13:0]};
```

The remaining 31 output words contain each remaining column address and the
timeout TDC value:

```verilog
timeout_tdc = {7'd0, tdc_time_out[6:0]};
rdata       = {13'd0, column[4:0], timeout_tdc[13:0]};
```

The active testbench drains those 32 words using:

```verilog
for (i = 0; i < 32; i = i + 1) begin
  wb_read32(ADDR_MATCH, rdata);
end
```

### Compute Test 1 Before Reconfiguration

Before runtime reconfiguration, `tdc_time_out` is `7'd32`, so timeout columns
return:

```verilog
timeout_tdc = {7'd0, 7'd32} = 14'h0020;
```

The testbench programs column `7`:

```verilog
wb_write(make_packet(2'b11, 5'd10, 5'd7, 1'b0, 1'b0, 8'h55));
wb_write(make_packet(2'b11, 5'd11, 5'd7, 1'b0, 1'b0, 8'h66));
wb_write(make_packet(2'b00, 5'd12, 5'd7, 1'b0, 1'b0, 8'h77));
```

That means:

```text
row 10, column 7 is SET
row 11, column 7 is SET
row 12, column 7 is RESET
```

Then it sends three compute packets:

```verilog
wb_write(make_packet(2'b10, 5'd10, 5'd7, 1'b0, 1'b0, 8'h08));
wb_write(make_packet(2'b10, 5'd11, 5'd7, 1'b0, 1'b0, 8'h07));
wb_write(make_packet(2'b10, 5'd12, 5'd7, 1'b0, 1'b0, 8'h08));
```

The selected column is `7`. Since rows 10 and 11 are SET, and row 12 is RESET,
only the first two compute data bytes contribute:

```verilog
compute_tdc = 8'h08 + 8'h07 = 14'h000F;
```

The first compute read should therefore contain:

```verilog
{13'd0, 5'd7, 14'h000F}
```

The remaining 31 reads should return columns:

```text
0, 1, 2, 3, 4, 5, 6, 8, 9, ..., 31
```

with:

```verilog
tdc_value = 14'h0020;
```

### Compute Test 2 After Reconfiguration

After runtime reconfiguration, `tdc_time_out` is `7'd40`, so timeout columns
return:

```verilog
timeout_tdc = {7'd0, 7'd40} = 14'h0028;
```

The testbench programs column `8`:

```verilog
wb_write(make_packet(2'b11, 5'd13, 5'd8, 1'b0, 1'b0, 8'h21));
wb_write(make_packet(2'b11, 5'd14, 5'd8, 1'b0, 1'b0, 8'h22));
wb_write(make_packet(2'b11, 5'd15, 5'd8, 1'b0, 1'b0, 8'h23));
```

That means rows 13, 14, and 15 are all SET at column `8`.

Then it sends three compute packets:

```verilog
wb_write(make_packet(2'b10, 5'd13, 5'd8, 1'b0, 1'b0, 8'hFF));
wb_write(make_packet(2'b10, 5'd14, 5'd8, 1'b0, 1'b0, 8'h12));
wb_write(make_packet(2'b10, 5'd15, 5'd8, 1'b0, 1'b0, 8'h04));
```

The selected column is `8`. Since all three selected cells are SET, all three
compute data bytes contribute:

```verilog
compute_tdc = 8'hFF + 8'h12 + 8'h04 = 14'h0115;
```

The first compute read should therefore contain:

```verilog
{13'd0, 5'd8, 14'h0115}
```

The remaining 31 reads should return columns:

```text
0, 1, 2, 3, 4, 5, 6, 7, 9, ..., 31
```

with:

```verilog
tdc_value = 14'h0028;
```

## Why There Is A Delay Before Wishbone Read

The model does not produce read data immediately when the READ command is
written. The READ command is accepted into a command FIFO. The engine later
processes it, waits the configured `READ_DELAY`, then pushes a result into the
response FIFO.

That is why the testbench uses delays such as:

```verilog
repeat (1000) @(posedge wb_clk_i);
```

before reading multiple queued response words.

If a Wishbone read is issued before response data is ready, the model does not
ACK yet. The `wb_read32` task waits at:

```verilog
wait(wbs_ack_o);
```

until the response appears.

## Reading And Decoding rdata

Each Wishbone read captures:

```verilog
reg [31:0] rdata;
```

Normal TDC output format:

```verilog
rdata = {13'd0, column[4:0], tdc_value[13:0]};
```

Decode:

```verilog
read_column = rdata[18:14];
tdc_value   = rdata[13:0];
```

Example print:

```verilog
wb_read32(ADDR_MATCH, rdata);
$display("rdata=%08h column=%0d tdc=%04h",
         rdata, rdata[18:14], rdata[13:0]);
```

## SET/RESET State From rdata

Using the initial configuration:

```verilog
SET midpoint   = 14'h0280;
RESET midpoint = 14'h0060;
threshold      = 14'h0170;
```

Classify a read result:

```verilog
if (rdata[13:0] >= 14'h0170)
  $display("Cell state looks SET");
else
  $display("Cell state looks RESET");
```

With the second runtime configuration:

```verilog
TARGET2_SET1   = 16'h0500;
TARGET2_SET2   = 16'h0400;
TARGET2_RESET1 = 16'h0100;
TARGET2_RESET2 = 16'h0180;
```

The midpoints are:

```verilog
SET midpoint   = 14'h0480;
RESET midpoint = 14'h0140;
threshold      = 14'h02E0;
```

Use the threshold that matches the currently active configuration.

## Notes About Existing Helper Code

The testbench contains helper functions for expected-value math and a
`check_read` task, but the active stimulus sequence uses direct `wb_read32`
transactions. The direct read style is:

```verilog
wb_read32(ADDR_MATCH, rdata);
```

This means the testbench reads and captures the value. It does not need to
compare against an expected value unless you explicitly call `check_read`.

## Common Mistakes

- Do not skip the first three configuration writes.
- Do not expect SET or RESET to return a TDC result directly.
- Send a READ command before expecting a TDC response.
- Send three COMPUTE packets before expecting compute response data.
- Perform 32 Wishbone reads after each compute operation.
- Remember that the first compute read is the selected column result.
- Remember that the remaining compute reads return timeout TDC values.
- Use `wb_read32(ADDR_MATCH, rdata)` to read the response FIFO.
- Decode `rdata[13:0]` as the TDC value.
- Decode `rdata[18:14]` as the returned column.
